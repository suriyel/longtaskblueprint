**语言规则**：用中文（简体）回复用户。所有生成的文档、报告和面向用户的输出用中文。Skill 名称、代码标识符、JSON 字段名保持英文。

# 需求打磨与 SRS 生成

输入：用户提供的需求文档（可能已近乎详细设计）+ 现有代码库上下文（若存量）。
动作：直接从用户输入提取候选 FR，交叉对照存量 ESI，用 EARS 改写并定量，单轮批准。
不做多轮挖掘式澄清，不做复杂度打分路由。

<HARD-GATE>
在 srs.md 获得用户批准（首行状态置「已批准」并 git commit）之前：
- 仅可写入 `**状态**: 待审批` 草稿，不可把状态置「已批准」、不可 git commit
- 不可调用任何设计 skill、实现 skill、编写任何代码、搭建任何项目、或采取任何设计/实现行动
- 整份一次性呈现并单次批准，不按章节/分段征求逐节反馈
每个项目都适用，无论看起来多简单。
</HARD-GATE>

## Step 0 — 下游打回处理

本节点可能因下游 **BDD 节点反推出需求缺口**而被打回重跑：

- **被打回（会话开头带整改要点）**：其内容即「待澄清清单」（逐条 `FR-xxx + 缺什么 / 未提及的行为`）。本轮**只就清单内的 gap 工作**：
  - 跳过全量打磨，直接进 Step 3 Gap Fill，用 `AskUserQuestion`（≤4 问/轮）仅就这些 gap 向用户澄清；
  - 把澄清结果**增量合并**进既有 `{{HARNESS_MEMORY_DIR}}/plans/srs.md` 对应的 FR / AC / CON（缺 AC 补 AC、未提及的补新 FR 并编号、不明确的定死取值/默认值/错误处理）；
  - **不重做、不重写**已批准的其余章节；保留首行 `**状态**: 已批准`，Git commit 增量改动；
  - 完成后照常推进（advance ok）。无需再跑 Step 1~Step 8 的全量流程。
- **未被打回**：属用户从 0 启动或常规续写，照常从下面的 Step 1 全量执行。

## Step 1 — 读上下文

1. **优先 Read `{{HARNESS_MEMORY_DIR}}/intent/user-original-intent.md`**。若用户在本会话另行上传 / 粘贴更详细的需求文档，与原话一并阅读。逐段标注：哪些是目标声明、哪些是行为规约、哪些是约束、哪些是实现偏好。
2. 检查 `{{HARNESS_MEMORY_DIR}}/notes/rules/`（若存在且已填充）— scan 节点提取的代码库约定：
   - `coding-constraints.md` — 二方件（内部强制库）约束
   - `build-and-compilation.md` — 构建系统约束
   - `coding-style.md` — 风格约束
3. 模板加载延后到 Step 1.2 — 由档位评估结果决定填写力度（模板只有一个 `{{SHARE-REFERENCE}}/srs-template.md`，空洞章节按需省略或标 `[不适用]`）。

## Step 1.2 — 输入成熟度评估（内部，不与用户交互）

按下表评估用户输入文档成熟度，结果决定 Step 1.5 / Step 2 / Step 7 Group P 的处理力度（`maturity_level`）：

| 档位 | 信号 | 处理力度 |
|------|------|---------|
| **L1 口语化** | 含口语词（"烦"/"卡"/"很"/"不要太"/感叹号）；硬精确密度 < 5/千字；价值/动机叙述为主 | 全套吸收-重写：领域语言重述、剥离偏好-约束、消化所有原话；SRS 全章节填写（§1.3 问题陈述 + §3 用户画像 + 用例视图等）|
| **L2 半成品规约** | 硬精确密度 5-30/千字；含具体参数名 / 文件路径 / 类型范围 / 默认值 / 枚举值；混合"功能说明 / 详细需求"段 | 意图段吸收（§1 范围 1-2 句）；**硬精确条目 1:1 锁定不重写**；空洞章节（§1.3 / §3 用户画像 / §3.1 用例视图等）标 `[不适用]` + 附理由 |
| **L3 完整规约** | 硬精确密度 > 30/千字；已分章节、含完整 FR/CON/接口规约、近规范范式 | 结构化映射：直接归类，仅做缺项 / 重复 / 冲突检测；空洞章节处理同 L2 |

**判定方法**：
1. 全文 grep 硬精确标识密度（camelCase / snake_case / `*.xml` / `*.json` / `[N, M]` / `default: N` / 枚举 `{a, b, c}`）
2. 全文 grep 口语词密度（"烦"/"卡"/"很"/"不要太"/"搞个"/感叹号）
3. **降档规则**：口语词密度 > 3/千字 → 强制降到 L1（混合输入按低档处理，避免漏剥离偏好）

**grep 对象**：包含 `{{HARNESS_MEMORY_DIR}}/intent/user-original-intent.md` 全文 + 用户在会话内补充的需求文档。档位评估的输入是用户**原话**，不是任何二次产物（不要拿尚未生成的 SRS 草稿评档）。

**模板加载**：
- 用户指定模板路径优先；否则加载 `{{SHARE-REFERENCE}}/srs-template.md`
- 验证 `.md` 且含至少一个 `## ` 标题

**输出**：`{maturity_level: L1|L2|L3}`，留主 agent 上下文驱动后续 step。Step 8.1 落盘时写入 SRS 元数据 `输入档位` 字段。

**反模式**：
- 跳过本评估、按 L1 处理所有输入 — L2/L3 输入会被强制套上空洞章节（§1.3 项目动机 / §3 用户画像 / §3.1 用例视图等）并硬凑内容
- L2/L3 输入硬填空洞章节而非标 `[不适用]` — 无意义内容增加 reviewer 审查噪声

## Step 1.4 — Brownfield 检测（内部，不与用户交互）

判定为存量系统的信号（满足任意 2 个即 `is_brownfield = true`）：
1. `{{HARNESS_MEMORY_DIR}}/notes/rules/` 存在且含 ≥1 个非桩 `.md`
2. `{{HARNESS_MEMORY_DIR}}/plans/codebase-scan.md` 存在
3. 用户描述含存量语言（"现有"、"已有"、"扩展"、"修改"、具体模块/文件路径）
4. Git 历史 ≥5 commits 且源文件 >3 个

内部记录 `is_brownfield` 与置信度，不询问用户。

## Step 1.5 — 语义吸收笔记（内部，不落盘，不与用户交互）

进入 Step 1.6 / Step 2 之前，主 agent 必须按 Step 1.2 评定的档位建立吸收笔记。
**纯认知操作**，无文件、无 SubAgent、无 schema；目的是阻断"扫读完就开写 EARS"的翻译式路径。

### 笔记项（按档位裁剪）

| # | 笔记项 | L1 | L2 | L3 | 内容要求 |
|---|--------|----|----|----|---------|
| A | 项目动机 | ✅ | ⚪ | ❌ | 1-3 句话领域语言重述用户想解决什么；驱动 §1.3（L1 输入填写；L2/L3 可整节标 `[不适用]`）|
| B | 领域实体清单 | ✅ | ✅ | ⚪ | 名词清单 + 用户隐含定义（含技术术语） |
| C | 关键操作/事件清单 | ✅ | ⚪ | ❌ | 动词清单 + 触发条件 |
| D | 偏好 vs 约束剥离 | ✅ | ⚪ | ❌ | "用 Redis"/"用 React" → ASM；"5ms 响应" → CON；混合句必须分离 |
| E | 隐含假设 / 待澄清 | ✅ | ✅ | ✅ | 用户写了 X 但没说 X 在 Y 情况下如何 — 喂 Step 3 Gap Fill |
| F | ESI 冲突点 / 复用点 | brownfield | brownfield | brownfield | 与 `{{HARNESS_MEMORY_DIR}}/notes/rules/` 或 codebase-scan 报告交叉对照（参见 `reference/brownfield-adaptation.md`） |
| **G** | **硬精确条目清单** | ⚪ | **✅** | **✅** | **参数名 / 字段名 / 文件路径 / 类型 / 范围 / 默认值 / 枚举值 / ID 命名 — 锁定 1:1 转写，禁改写禁稀释** |

### 笔记 G 详细规则（L2/L3 关键）

笔记 G 的条目格式：

| 标识 | 类型 | 来源 L行号 | 锁定字段 |
|------|------|-----------|---------|
| `UrbanBuildingSearchRadius` | enum | L7 | 取值: {60,80,100,120,140,160,180,200,'自适应'}; 默认: <未指定 → Gap Fill> |
| `UrbanPolygonBufferDistance` | int | L8 | 范围: [0, 5000]; 默认: 50 |
| `buildingsDistinguish_parameter.xml` | file path | L6 | 参数文件归属 |

笔记 G 条目在 Step 2 重写时**原样进入** EARS / AC / IFR，禁"领域化美化"。

**反模式**：
- 把 Step 1.5 写成原文摘要 — 摘要仍是翻译。吸收要求**用自己的话重述**领域知识，丢掉用户的语调与措辞
- L2/L3 输入跳过笔记 G，对参数名做"领域化美化"（`UrbanBuildingSearchRadius` 改成"城市建筑搜索半径配置项"）— 信息损失，下游失去技术契约依据

## Step 1.6 — 针对性存量探索（仅 brownfield）

**触发**：`is_brownfield = true` 且用户描述包含具体功能 / 领域区域 / 模块关键词。
**跳过**：新建项目；或用户描述过于模糊无法推导 focus。

执行：
1. 从用户描述提取领域关键词（如"认证"/"支付"/"数据管道"），推导 `--focus` 维度。
2. 推断 `--path`（若用户提及具体目录）。
3. 深度默认省略（让 explore 按 LOC 自检）；用户输入明显窄范围可降。
4. 直接执行代码库探索（不 spawn SubAgent）：

   加载并执行技能 explore-guide，按其流程顺序执行全部探索阶段（Phase 1 结构扫描 → [按深度决定是否进入 Phase 2 维度分析] → 综合 → 输出）。

   参数：
   - Depth: {省略或推导}
   - Focus: {推导的维度}
   - Path: {推导的路径或 "."}
   - User question: "{用户描述摘要}"
   - output_path: `{{HARNESS_MEMORY_DIR}}/notes/codebase-research.md`
   - rules_dir: `{{HARNESS_MEMORY_DIR}}/notes/rules/`
   - report_template: `{{SHARE-REFERENCE}}/explore-report-template.md`

   补充上下文（只读）：`{{HARNESS_MEMORY_DIR}}/plans/codebase-scan.md`（若存在）作为 baseline。

5. 若返回有用结果 → 按 `reference/brownfield-adaptation.md` §A 构建 ESI 表（内部态）。
6. BLOCKED / 无有用结果 → 静默跳过，仍照常进入 Step 2。此步非阻塞。

## Step 2 — 按 SRS 范式重写（内部，不与用户交互）

基于 Step 1.5 吸收笔记 + Step 1.2 档位 + 选定模板，按 SRS 范式（FR / CON / ASM / IFR / EXC）**重新表述**领域知识。
**不是**翻译用户原文为 EARS — 翻译式产物已被打磨度审查门（Step 7 Group P）拒收。

### L1 口语化输入处理

1. **重写**：以 Step 1.5 笔记 B（实体）/ C（操作）为基础构造候选条目，用领域语言书写。
   **禁止保留用户原词作为 EARS 主体**。原文段落仅作为来源行号引用，不作为 EARS 文本来源。
2. **分类**：按 ID 前缀归入 FR / CON / ASM / IFR / EXC（详见 Step 4.1）。
   实现细节（笔记 D 已剥离）归 ASM / CON / IFR，**不**作为 FR。
3. **交叉对照 ESI**（仅 brownfield，笔记 F）：标注 `NEW` / `MODIFY` / `EXTEND` / `REUSE`（REUSE → ASM-xxx — 见 `reference/brownfield-adaptation.md` §C）
4. **§1.3 项目陈述写作**：基于笔记 A，2-3 段领域语言叙述；**禁原话痕迹**（Group P1/P2 审查）

### L2 半成品规约处理 ★

1. **§1 范围**：基于笔记 A，1-2 句话简述；保留必要技术上下文（如"在既有 X 模块新增 Y 配置"）。**§1.3 问题陈述 / §3 用户画像 / §3.1 用例视图**等空洞章节标 `[不适用]` + 附一句理由（如"已给硬精确规约，无需 5-Whys"）
2. **硬精确条目转写**：笔记 G 每个条目原样进入 EARS / AC / IFR：
   - 参数配置 → FR：`The system shall expose <参数名> as <类型> config in <文件>`；AC 列范围/默认值/枚举/错误处理
   - 文件路径 / XML 元素 → IFR：`The system shall accept <节点名> as path nodes in <文件>`
   - **EARS / AC 中保留参数名、文件名、类型范围原样** — 这些是技术契约不是"原话"
3. **不做偏好剥离**：L2 输入通常无 React/Redis 类偏好混杂；笔记 D 可空
4. **缺项检测**：对照笔记 G 检查每条参数是否齐全（类型 + 范围 + 默认值）；缺默认值 / 缺范围 → Step 3 Gap Fill 询问

### L3 完整规约处理

1. **结构化映射**：用户文档已分章节 / 已有 ID → 直接按 ID 归类到 SRS 对应章节，**不重写句子**
2. **冲突检测**：检查重复 ID、矛盾约束、不完整 EARS（缺主语 / 缺触发条件）
3. **§1 范围**：直接复用用户原文章节标题（若有）

### 识别 gap（所有档位）

基于 Step 1.5 笔记 E，列必须向用户确认的空白：
- **冲突**：用户输入内部矛盾，或用户输入 ↔ ESI 冲突
- **缺 AC**：用户只描述成功路径，未提错误/边界
- **未声明约束**：用户输入暗示但未明说的硬性限制
- **范围边界模糊**：用户描述边界与 `{{HARNESS_MEMORY_DIR}}/notes/rules/` 或 ESI 发生拉扯的区域
- **L2/L3 缺项**：硬精确条目缺类型 / 范围 / 默认值

### 原文引用规则（强制 — Step 7 Group P 审查）

| ✅ 允许位置（所有档位） | 形式 |
|---|---|
| §2 术语表"用户用语"列 | 表格单元 |
| AC 中具体场景示例 | Given/When/Then 行内引号 |
| 必要章节脚注 | `> 用户原文（L42-45）：...` 仅当吸收后表达可能与原意有偏差时 |

| ❌ 禁止位置 | L1 反例 | L2/L3 反例 |
|---|---|---|
| EARS 主体含口语 | `The system shall 让用户登录` | `The system shall 烦死人地推送` |
| §1 / §1.3 含纯口语 | "用户希望搞个系统" | "搞一个 Web 系统" |
| 章节标题用口语 | `### 让用户高兴` | `### 烦死的 Excel` |
| 引用未带源行号 | `> "支持多端"` 无 (L?) | 同 L1 |

**L2/L3 显式允许**：EARS / AC / IFR 主体中出现 camelCase / snake_case 标识、文件路径、数值范围、枚举值、XML 元素名 — 这些是**技术契约**不是"原话"，不计入 Group P 原话痕迹扫描。

产出（内部态）：候选条目草表（按档位处理）+ §1 / §1.3 草稿 + gap 清单。

## Step 3 — Gap Fill（单轮 AskUserQuestion，≤4 问）

**仅**针对 Step 2 标注的 gap 提问。不重复用户已在输入文档中写明的内容。

若 brownfield：使用 `reference/brownfield-adaptation.md` §B 的校核式问句模板（"我从输入提取到 FR-X。它与 ESI 现存 Y 是替换/扩展/并行？"）。

**Gap Fill 输出**：用户回答整合到候选 FR 草表与 §1.2 排除范围 / §6 约束 / §7 假设。

**升级逃生门**：若 Step 2 产出的候选 FR ≤ 3 且无明确目标（用户输入过于模糊）→ 告知用户"输入不足以打磨，切换到挖掘模式"→ 进入 3-5 轮扁平提问（每轮 ≤4 问，覆盖问题陈述、角色、范围边界、能力清单、错误路径）。挖掘完成后回到 Step 4。

## Step 4 — EARS 编写 + 图表

### 4.1 分类与 ID

| 类别 | ID 前缀 | 描述 |
|------|---------|------|
| 功能 | FR-001 | 可观察的系统行为 |
| 约束 | CON-001 | 限制解决方案空间的硬性限制 |
| 假设 | ASM-001 | 假定为真的信念；记录失效风险 |
| 接口 | IFR-001 | 外部系统契约 |
| 排除 | EXC-001 | 明确不在范围内 |

### 4.2 EARS 模板

| 模式 | 模板 | 适用场景 |
|------|------|---------|
| 普遍型 | The system shall `<action>`. | 始终生效 |
| 事件驱动型 | When `<trigger>`, the system shall `<action>`. | 响应事件 |
| 状态驱动型 | While `<state>`, the system shall `<action>`. | 模式/状态相关 |
| 异常行为型 | If `<condition>`, then the system shall `<action>`. | 错误处理 |
| 可选型 | Where `<feature/config>`, the system shall `<action>`. | 可配置能力 |

每个 FR 配：至少一个 Given/When/Then 验收标准；MoSCoW 优先级；来源（用户输入段落 / ESI gap / 走查发现）。

### 4.3 图表

- **用例视图**（§3.1）：`graph LR`，角色为 `Actor((Name))`，FR-xxx 作为 `subgraph System Boundary` 内的节点，角色→用例绘制边。
- **流程图**（§4.1）：每个有 3+ 个顺序步骤或分支的功能区域一个 `flowchart TD`。起止 `([label])`，决策 `{condition?}`，标注 `-- YES -->` / `-- NO -->`。

### 4.4 语言精炼

- 一句能说清的不用两句；一个词能表达的不用短语
- EARS 声明直达行为，不加修饰语与背景解释
- Given/When/Then 每段 ≤1 行
- 排除"为了...所以..."式动机（动机归 §1.3）
- 术语表仅收录有歧义的术语
- 存量：不重复 ESI 已确立能力，仅写增量

## Step 5 — 粒度分析（双向定量）

**粒度主轴 = 实现单元**：每个 FR 应对应**一个 Worker 会话能独立做完、且能被一组连贯验收标准独立验证的特性**，经验体量约 1,000 行实现代码（不含 UT），约耗一个 Worker 会话 ~50% 上下文。

判断依据**优先级**（高 → 低）：
1. **能否独立验证为一个连贯特性**（主判据）：同一接口 / 组件 / 实体 / 角色上的一组行为，放在一起才构成一个可交付、可端到端验证的单元 → 应为 1 个 FR。
2. **实现体量是否接近一个会话单元**（主判据）：明显远超（一个会话做不完）→ 偏大；一组 FR 合计才够一个会话单元 → 偏小。LLM 估 LOC 不准，故只做**数量级**判断（远超 / 适中 / 远低），不抠精确行数。
3. **AC 数量分布**（次级粗信号，**不单独构成拆 / 合理由**）：仅当与判据 1/2 一致时作佐证。AC 多 ≠ 该拆（可能都是同一行为的变体）；AC 少 ≠ 该合（可能是高复杂度核心算法）。

### 5.0 配置

| 上下文窗口 | 每 FR 参考 AC 区间（粗信号） | 单元体量参考 |
|------------|------------------------------|--------------|
| ≤ 200K tokens | 3-12 | ~1,000 行实现代码 |
| > 200K tokens | 5-20 | ~1,000 行实现代码 |

> AC 区间是**软参考**，非硬阈值。落区间外不自动触发拆 / 合，须先过判据 1/2。本表上限（12/20）仅用于**校核拆分**（超上限才回看是否真有多个独立行为），**不**作「AC 多就拆」依据。

### 5.1 过大检测（G1-G6）— 拆分

| # | 启发式 | 触发信号（均须「拆后每子项仍可独立验证」） |
|---|--------|---------------------------------------------|
| G1 | 多角色 | 2+ 角色执行**职责不同**的操作 |
| G2 | CRUD 捆绑 | 增删改查为单一需求，**且该实体 CRUD 合计预计 ≥ 一个会话单元**（小实体的全 CRUD 本就是一个特性，不拆）|
| G3 | 行为路径分化 | 同一 FR 覆盖 **2+ 个语义不同的行为路径，且各路径自身就接近一个会话单元**（同一行为的变体——如一组同类指令、同一校验的多种非法输入——**不**触发）|
| G4 | 跨层关注 | 后端 + UI 在同 FR，且各层各自可独立交付验证 |
| G5 | 多状态行为 | 3+ 不同状态 / 模式，**且各状态分支逻辑独立、合计 ≥ 一个会话单元**（同一行为的有限参数取值——如方向 / 档位枚举——不拆）|
| G6 | 时序耦合 | 触发事件 + 延迟 / 定时后果，且前后两段可分别验证 |

候选 → 识别原子行为，单一职责测试，保持可追溯（FR-003 → FR-003a/b），重新验证子项。

> **注**：G3 不再以「AC 条数」为触发依据——AC 多寡只是粗信号（见 Step 5 抬头判据 3）。是否拆只看「是否多个各自接近一个会话单元的独立行为路径」。

### 5.2 过小检测（S1-S5）— 合并

| # | 启发式 | 信号 | 操作 |
|---|--------|------|------|
| S1 | 微小新增 | 单字段 / 常量 / 配置，≤1 AC | 合并到父实体 FR |
| S2 | 单 AC 无负例 | 仅 1 AC 且无错误 / 边界 | 充实 AC，或合并到共享实体 / 端点的相关 FR |
| S3 | 纯数据回显 | 无转换地显示另 FR 产物 | 作为垂直切片合并 |
| S4 | 仅配置 / 脚手架 | 环境 / 依赖安装，无业务逻辑 | 合并为 Foundation FR |
| **S5** | **同源兄弟聚合** | 2+ 个 FR **同时满足**：① 共享同一接口 / 入口（同一命令分发、同一端点、同一类的方法族）；**且** ② 同一主角色；**且** ③ 操作同一领域实体 / 状态；**且** ④ 每个个体实现体量都**远低于**一个会话单元（数量级判断）| 聚合为**一个「特性族」FR**：EARS 用普遍型 / 事件驱动型概括共同行为（如「处理 `<接口>` 上的 `<指令集>`」），每个原兄弟 FR 的具体行为**完整保留为一条 AC** |

**合并规则**：
- 内容保留强制 — 被吸收 FR 的 EARS + 全部 AC + 描述文本完整并入主 FR，不摘要化
- **S5 不受 S1-S4 的「个体琐碎」前置约束** — S1-S4 只抓个体即琐碎（≤1 AC / 纯回显 / 纯配置）的 FR；**一组各自合法（各有 2-3 个 AC）、个体不琐碎、但同源且都远低于一个会话单元的兄弟 FR，正是 S5 专门覆盖的盲区**
- 合并 FR 必须共享主角色 + 功能区域（S5 的 ①②③ 即此约束的精确化）
- 合并后若 AC 超 5.0 上限 → 按**行为接缝**（而非「逐条指令」）重新评估拆分，避免拆回聚合前形态
- 合并后按顺序重编 FR ID 并更新全部交叉引用（用例视图、流程图、可追溯矩阵）；**重编号须保持三位零填充**（FR-001），下游 gate_init 机检该格式

> **S5 落地示例（泛化，跨领域）**：
> - 同一控制接口的一组指令（转 / 移 / 停）→ 1 个「指令执行」FR；
> - 同一配置组件的一组 setter（设 A、设 B、设阈值）→ 1 个「配置管理」FR；
> - 同一前端容器的一组视图切片（渲染、操控面板、提示弹窗，且都依附同一后端能力、无独立后端逻辑）→ 1 个「前端交互」FR。

### 5.2bis 聚合 pass（下限制衡 — 强制跑一趟）

在 5.1 全部拆分候选应用之后，**无条件**对当前 FR 全集自下而上跑一趟 S5 聚合扫描——这是破「先拆、子项各 ≥2 AC 后又合不回来」死循环的关键。

1. 对每组共享接口 / 实体 / 角色的 FR，套用 S5 判据。
2. **数量级红线**：若一组 FR 合计 ≈ 一个会话单元却被拆成 ≥3 份（估计平均体量 < 单元的 ~1/3），**强制生成聚合提议**进 5.3。
3. **G 与 S5 同时命中同一组 FR 时**：不再机械「G 优先」，而**以「最终是否构成一个可独立验证的会话单元」为准**——若拆分会产生一堆相互依赖、无法独立验证的碎片，则不拆 / 拆后立即被本 pass 聚合回去。

### 5.3 决策阈值

聚合类提议（S5 / 5.2bis）与拆分类提议**分开计数、分屏呈现，且聚合在前**（让用户先看到「收敛后」形态，避免在一堆碎 FR 上做拆分微调）。

| 候选数量 | 操作 |
|---------|------|
| 0 | 跳过 |
| 1-3 | 自动应用，内联呈现理由（聚合理由须列出共享的接口 / 角色 / 实体）|
| 4+ | AskUserQuestion 批准（聚合提议与拆分提议分两屏，聚合在前）|

### 5.4 FR 粒度确认（强制）

所有拆分 / 合并 / 重编号后，呈现最终 FR 表供用户确认：

| FR ID | 标题 | 共享接口 / 角色 / 实体 | AC 数 | 体量量级（远低 / 适中 / 偏高）| 变更？ | 备注 |
|-------|------|------------------------|-------|------------------------------|--------|------|
| | | | | | 从 FR-X 拆分 / 吸收 FR-Y / 未变更 | |

表下附**汇总信号行**（强制填写）：
> 当前 {N} 个 FR，体量量级分布：{远低 x / 适中 y / 偏高 z}。若「远低」占比高 → 提示存在过度拆解，建议复核 S5 聚合。

问（**中性措辞，不预设答案**）："目标：每个 FR ≈ 一个 Worker 会话能独立做完的特性（~1,000 行实现代码，不含 UT），合并 FR 保留被吸收内容完整并集。当前 {N} 个 FR 中有 {x} 个体量量级偏低。是否需要进一步聚合，还是确认当前粒度？"
- AskUserQuestion 选项：`[按建议聚合到 {M} 个] / [确认当前 {N} 个] / [我来指定如何合并]`；当「远低」占比高时「聚合」列为首选项（不在「确认」上加「推荐」预设）。
- 确认 → Step 5.5
- 请求调整 → 应用、重编号、重新呈现

即使 0 候选（无拆分 / 合并）也呈现 FR 列表供确认。

### 5.5 单轮模式确认（强制）

通过 AskUserQuestion：

> "最终 FR 列表 {N} 个。**单轮模式可用**：全部 FR 在本轮（wave 0）实现，不延迟。每个 FR 映射到一个功能；Worker 每会话一个功能。**若某 FR 预计远超一个会话单元 → 进一步拆分；若多个 FR 合计才够一个会话单元 → 回 5.2bis 聚合。**
> [启用单轮模式] / [跳过 — 进入延迟分析]"

- 启用 → SRS 元数据记 `Single-Round: Yes`。Step 6 仍跑但只呈现确认摘要。
- 跳过 → Step 6 正常延迟分析。

## Step 6 — 范围适配与延迟

评估所有需求是否属于本轮。标准：优先级、依赖、完整性、风险、范围预算。

规则：
- Must 级 FR 不自动延迟
- 若 FR-X 依赖 FR-Y，两者同进同出
- 延迟积压保留 EARS + AC 供 increment 拾取

若有延迟 → 生成 `{{HARNESS_MEMORY_DIR}}/plans/srs-deferred.md`。

## Step 7 — 质量自检

完成 SRS 草稿前自检：FR / NFR / 接受准则 / 约束 / 假设五段必须齐全，每个 FR 必有 ≥ 1 个 Given/When/Then 接受准则；标识符 1:1 匹配用户原文。

**注**：harness 在本节点结束后会自动跑 `gate_srs.cjs` 硬门做存在性 + 段落完整性机器校验；本步骤是 LLM 自检层。

## Step 8 — 落盘草稿、呈现、单次批准、提交

### 8.1 落盘整份草稿（未 commit）

读取 Step 1.2 加载的模板（`{{SHARE-REFERENCE}}/srs-template.md` 或用户自定义），**一次性装配整份 SRS 落盘为草稿**：
1. 保留标题结构
2. 每个标题下替换指导文本为 Step 2–7 产出的内容（整份一次性装配，不在对话中按章节呈现）
3. 顶部元数据若无则加：`日期`、**`状态`: 待审批**、`标准`（ISO/IEC/IEEE 29148）、`输入档位`（L1/L2/L3）、`Single-Round: Yes/No`（若适用）
4. L2/L3 输入下的空洞章节：整节标 `[不适用]` + 附一句理由（如"已给硬精确规约，无需 5-Whys"）
5. 产出内容若无匹配章节：追加为"附加说明"

保存整份到 `{{HARNESS_MEMORY_DIR}}/plans/srs.md`，延迟积压一并保存。**不执行 git commit。**

### 8.2 呈现待审批

落盘后输出文本（非 AskUserQuestion）告知用户：文件路径 + 需阅读整份 SRS 全文 + 阅后回对话做一次性整体批准。**不分段 / 按章节呈现，不逐节征求反馈。**

### 8.3 单次批准与提交

通过**一次** `AskUserQuestion` 征求整份批准：`[批准] / [要修改（说明改哪里）]`。
- **批准** → 把 srs.md 首行 `**状态**: 待审批` 改为 `**状态**: 已批准`；`git commit`；照常推进（advance ok）。
- **要修改** → 按用户说明修订**整份**草稿后重新落盘（覆盖同一文件，仍 `**状态**: 待审批`）+ 重新呈现 + 重新征求一次性批准（仍是整体单次，不退化为逐章）。

**收到「批准」前不可把状态置「已批准」、不可 git commit、不可推进到设计阶段。**

## 反模式

| 合理化 | 正确回应 |
|--------|---------|
| "这太简单了，不需要 SRS" | 打磨模式本身就是简单路径。短 SRS 可接受。 |
| "用户输入是原文，直接转设计" | 用户输入混合了目标/行为/实现偏好。SRS 增加分类、完整性、可测试性。 |
| "让我先问用户想要什么" | 用户已写在输入文档。先读、先提取、再只问 gap。 |
| "L2 输入也应有完整 §1.3 项目动机" | L2 是实现规约式增量需求，§1.3 / 用户画像 / 用例视图 都是空洞填充。Step 1.2 评定 L2 → 整节标 `[不适用]` + 附一句理由。 |
| "把 `UrbanBuildingSearchRadius` 美化为'城市建筑搜索半径'更易懂" | 美化 = 信息损失。下游需 1:1 引用原标识。Group P5 严格审查白名单标识 100% 在 SRS 中体现。 |
| "翻译用户原话成 EARS 就行" | 翻译式产物 Group P 拒收。Step 1.5 吸收 → Step 2 用领域语言重写，丢掉用户语调与措辞。 |
| "ESI 已覆盖的能力仍写成 FR" | REUSE 转 ASM-xxx 或根本不记。FR 只写增量。 |
| "这个 FR 小但清晰，不合并" | S1-S5 启发式 — 过小 FR 浪费会话固定开销。 |
| "这些 FR 各自都清晰合法，不用合" | 个体合法 ≠ 粒度合理。同源（同接口 / 角色 / 实体）且都远低于一个会话单元的兄弟 FR，按 S5 聚合成特性族——碎 FR 会引发下游 BDD 场景爆炸与会话固定开销浪费。 |
| "所有需求都属于本轮" | Step 6 评估；延迟低优先级保留 EARS + AC 到积压。 |
| "用户输入暗示了设计方案就跳 SRS" | 需求定义 WHAT，设计 = HOW。两份都要。 |
| "无障碍不适用于此项目" | 任何用户界面都有 WCAG 2.1 AA 基线。 |
| "隐私在设计处理" | 隐私需求必须在 SRS — 数据模型与同意流程依赖它们。 |
